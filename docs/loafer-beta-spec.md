# loafer (beta) 仕様書・設計計画書

## 0. 仕様策定の目的

本書は、クリエーション・リサーチ習慣化アプリケーション **loafer (beta)** の初期仕様を定義し、プロトタイプ実装に入る前に未確定事項を洗い出すための設計計画書である。

loafer は、既存のアーティストが日々のリサーチを「格闘」として記録し、その身体的・物質的な出来事を論理的な「舞踊譜（SCORE）」へ変換・蓄積することを目的とする。

## 1. プロダクト・コンセプト

- **名称**: loafer (beta)
- **スローガン**: 身体を地平へ、格闘を譜面へ
- **対象ユーザー**: 日々の創作リサーチを身体・物質・言語の往復として記録したいアーティスト、振付家、パフォーマー、リサーチャー
- **コアバリュー**: 10分間の身体的リサーチを、APL / MAL 的な論理に基づく「マテリアルの変換」として記述し、習慣化させる。
- **設計哲学**: 人間を操作主体ではなく「地形」として捉え、紙や衣服、ポケット、重力、音、AI との交渉を「抵抗」として体験する。

## 2. 中心となる概念モデル

### 2.1 Body[]: 地形としての身体

ユーザーの身体は、意志を持つ中心主体ではなく、紙や重力や音が通過する地形として扱われる。

記録対象の例:

- 肩、腕、胸、腹、背中、脚などの接触可能な身体領域
- 衣服の素材、厚み、摩擦
- ポケット、襟、袖、裾などの隙間
- 紙が留まる、落ちる、折れる、滑る、埋まる場所

### 2.2 3.5 Dimension Pocket: 衣服と皮膚の間の隙間

ポケットは単なる収納領域ではなく、紙が潜在化し、再び抽出される中間次元として定義する。

記録対象の例:

- ポケットの位置
- 入口の広さ
- 深さ
- 素材の抵抗
- 取り出しやすさ
- 身体運動による紙の移動可能性

### 2.3 M_paper: アクターとしての紙

紙は道具ではなく、変形し、抵抗し、埋没し、抽出されるアクターとして扱う。

記録対象の例:

- サイズ
- 形状
- 折れ
- 破れ
- 湿り
- しわ
- 汚れ
- 身体や衣服への付着状態

### 2.4 CA: 共同主体としての AI

AI はユーザーを補助するだけでなく、セッション中に制約や摩擦を注入する共同主体として設計する。

CA の役割:

- セッション開始の儀式化
- 抵抗タスクの提示
- ユーザーの記述の再定義
- SCORE の生成支援
- セッション後のリフレクション対話

## 3. MVP のスコープ

beta の最初の MVP では、全機能を実装せず、以下に集中する。

1. 1分間のプロトタイプ・セッション
2. タップによる Extract（⊃）開始
3. 簡易タイマー
4. 手動入力による身体・紙・抵抗の記録
5. AI 風の固定テンプレートによる Friction Injection
6. セッション終了後の SCORE 自動生成
7. セッション履歴のローカル保存

## 4. オンボーディング仕様

### 4.1 初期登録

一般的なプロフィール登録ではなく、身体を地形として初期化する。

入力項目案:

- 表示名またはコードネーム
- Body[] の主な地形タグ
- 使用する衣服タイプ
- 使用する紙の種類
- 主に使う 3.5 Dimension Pocket の位置
- 1回の標準セッション時間

### 4.2 Body[] セットアップ

初期バージョンではカメラやセンサーによる自動スキャンは行わず、ユーザーが身体領域とポケットを手動で登録する。

将来的な拡張:

- カメラによる衣服・ポケット位置の記録
- 加速度センサーによるシェイク検出
- ジャイロセンサーによる Rotate 検出
- 音声入力による Potentialize 検出

### 4.3 3s Hold Initiation

セッション開始前に画面を3秒長押しすることで、仮想の紙をポケットから Extract（⊃）する。

MVP では以下の UI とする。

1. 「ポケットに紙が眠っている」という待機画面を表示する。
2. ユーザーが3秒長押しする。
3. 長押しが完了すると、セッションが開始される。
4. 最初の演算子として `⊃` を SCORE に記録する。

## 5. コア・ループ仕様

### 5.1 セッション単位

- MVP: 1分間
- beta 標準: 10分間
- 将来: 10分間を複数回繰り返すセット運用

### 5.2 アクションと MAL 演算子

| アクション | MAL 演算子 | 意味 | MVP 対応 |
| --- | --- | --- | --- |
| タップ | Extract (`⊃`) | 紙を潜在空間から顕在化させる | 対応 |
| シェイク | Vertical (`V/`) | 上下運動により紙を地形に従わせる | 将来対応 |
| 長押し | Delay | 落下や判断をためらわせる | 一部対応 |
| 音声 | Potentialize (`~`) | 音響的な場を作り紙を不在化させる | 将来対応 |
| 旋回 | Reverse (`⌽`) | 視線や身体を反転させ背面を顕在化させる | 将来対応 |

### 5.3 セッション中の記録

MVP ではセンサー自動記録よりも、ユーザーの短い入力と選択式ログを優先する。

記録項目:

- 起きた身体領域
- 紙の状態変化
- 抵抗の種類
- 失敗、落下、遅延、埋没の有無
- CA からの制約に対する応答

## 6. Friction Injection 仕様

### 6.1 基本方針

CA はユーザーの行為を滑らかに支援しすぎない。むしろ、一定の不自由さ、制約、再定義を与える。

### 6.2 MVP の抵抗テンプレート

初期実装では、ランダムまたは順番に以下のような固定プロンプトを提示する。

- 手を使わず、身体の地形だけで紙を移動させよ。
- 紙を守るのではなく、紙が落ちる条件を観察せよ。
- その紙を「大切なもの」ではなく「ゴミ」として扱え。
- 音が止まる直前の身体の遅延を記録せよ。
- いま起きた失敗を、成功よりも優先して SCORE に入れよ。

### 6.3 叙情性の剥奪

ユーザーが紙を過度に愛護的・象徴的に扱う場合、CA は紙を物質またはゴミとして再定義する。

例:

```text
CA: これは記憶ではない。これは紙である。
CA: これは赤ちゃんではない。これはゴミである。
CA: パージせよ。ただし、痕跡を記録せよ。
```

## 7. SCORE 生成仕様

### 7.1 SCORE の目的

SCORE はセッションの感想文ではなく、身体・紙・抵抗・時間の変換列である。

### 7.2 MVP の SCORE フォーマット

```text
SCORE_YYYYMMDD_HHMM ← ⊃(M_paper ∘ Body[region]) → Delay(event) → V/(state_change) → ~Trace(note)
```

### 7.3 SCORE 生成に使う入力

- セッション開始演算子
- 選択された Body[] 領域
- 紙の初期状態
- 紙の終了状態
- 抵抗テンプレート
- ユーザーの短いメモ
- 写真または痕跡情報

### 7.4 生成例

```text
SCORE_20260703_1042 ← ⊃(M_paper ∘ Body[chest/pocket]) → Delay(fall) → V/(crease, slide) → ~Trace(redness, torn_edge)
```

## 8. アーカイブ仕様

### 8.1 保存するデータ

- セッション時間
- SCORE

- Pocket 情報
- 紙の状態
- CA 抵抗テンプレート
- ユーザーメモ
- 写真パスまたは画像メタデータ

### 8.2 履歴画面

履歴画面では、日記ではなく SCORE の束として表示する。

表示案:

- 日付
- SCORE の短縮表示
- 抵抗タグ
- 変形タグ
- 写真サムネイル

## 9. 画面構成

### 9.1 MVP 画面

1. オンボーディング画面
2. Body[] / Pocket 登録画面
3. セッション待機画面
4. 1分間セッション画面
5. SCORE 生成画面
6. アーカイブ画面

### 9.2 セッション画面の UI 方針

- ゲーム的だが、達成感よりも「抵抗への応答」を強調する。
- ボタン文言は操作説明ではなく、儀式的な語彙を用いる。
- 操作ログは人間の意思ではなく、紙・身体・時間の関係として表示する。

## 10. 技術設計の初期方針

リポジトリの実装技術は未確定のため、現時点ではプラットフォーム非依存の設計とする。

候補:

- **Web / PWA**: 最速で検証しやすい。スマートフォンでも利用可能。
- **React Native / Expo**: センサー、カメラ、通知との相性がよい。
- **Flutter**: iOS / Android 両対応で UI 表現を作り込みやすい。

MVP 推奨:

- 最初は Web / PWA または Expo で実装する。
- センサー連携は後回しにし、手動入力で概念検証を優先する。
- SCORE 生成は初期段階ではテンプレート生成にする。

## 13. 現時点の推奨判断

未確定事項が多いため、最初の実装は以下を推奨する。

- プラットフォーム: スマートフォン向け Web / PWA
- センサー: MVP では未使用
- セッション: 1分間
- SCORE: APL / MAL 記法と自然言語説明を併記
- AI: 固定テンプレートの CA として開始
- 保存: ブラウザのローカルストレージ
- 写真: MVP では任意入力または後回し
- 共有: beta 後半まで保留

この方針により、哲学的な体験の核である「タップ＝抽出」「身体＝地形」「紙＝アクター」「AI＝共同主体」を最短で検証できる。

## 14. これから必要なステップの切り分け

### Step 1: 意思決定が必要な仕様を確定する

実装前に、体験の核に関わる選択だけを先に決める。

- 最初の対象環境を決める。
- MVP にセンサー、カメラ、音声を含めるか決める。
- SCORE を編集可能にするか、固定された痕跡として扱うか決める。
- CA の口調と抵抗の強度を決める。
- 写真保存と共有機能を MVP に含めるか決める。

### Step 2: MVP の仕様を最小単位に落とす

最初の実装対象を「1分間の Extract プロトタイプ」に限定する。

- 3秒長押しでセッションを開始する。
- `⊃` を最初の演算子として記録する。
- 1分タイマーを表示する。
- CA が1つの抵抗プロンプトを提示する。
- セッション後に短いメモを入力する。
- SCORE をテンプレートで生成する。
- ローカルに履歴を保存する。

### Step 3: 情報設計と画面遷移を作る

コードを書く前に、画面とデータの流れを固定する。

- オンボーディング画面
- Body[] / Pocket 登録画面
- セッション待機画面
- セッション実行画面
- セッション終了入力画面
- SCORE 表示画面
- アーカイブ画面

### Step 4: データモデルを定義する

実装で迷わないように、最小データ構造を先に決める。

- `BodyRegion`
- `Pocket`
- `PaperMaterial`
- `FrictionPrompt`
- `SessionLog`
- `Score`
- `TraceArchiveItem`

### Step 5: プロトタイプを実装する

最初は完成度よりも、体験の検証を優先する。

- UI フレームを作る。
- 3秒長押しを実装する。
- 1分タイマーを実装する。
- 抵抗プロンプトを表示する。
- セッション後入力を実装する。
- SCORE 生成関数を実装する。
- ローカル保存を実装する。

### Step 6: 体験テストを行う

通常の機能テストだけでなく、コンセプトが身体的に成立しているかを確認する。

- 1分間で「抽出」の感覚が立ち上がるか。
- CA の抵抗が単なる通知ではなく、行為を変えるか。
- SCORE が日記ではなく舞踊譜として読めるか。
- 入力項目が多すぎてリサーチを止めていないか。
- 紙の変形や失敗が記録対象として扱われているか。

### Step 7: 10分間 beta へ拡張する

1分間プロトタイプで核が確認できたら、標準セッションへ拡張する。

- セッション時間を10分へ変更する。
- 複数の Friction Injection を差し込む。
- 写真記録を追加する。
- アーカイブ一覧を改善する。
- 通知や習慣化の導線を検討する。

## 15. 次にやるべきアクション

次にやるべきことは、**MVP の対象環境を「スマートフォン向け Web / PWA」に仮決定し、1分間 Extract プロトタイプの実装チケットを作ること**である。

理由:

- 現時点でセンサーやカメラを入れると、哲学的な体験検証より技術検証が先行してしまう。
- Web / PWA であれば、スマートフォン上で身体・紙・ポケットの実験をすぐに試せる。
- 1分間に限定すれば、「タップ＝抽出」「CA＝抵抗」「SCORE＝変換列」が成立するかを最短で確認できる。

最初の実装チケット案:

```text
Title: Build 1-minute Extract prototype

Scope:
- Create minimal smartphone-first Web / PWA app shell.
- Add onboarding fields for Body[] region, pocket position, and paper type.
- Add 3-second hold initiation screen.
- Start a 1-minute session after successful hold.
- Show one CA friction prompt during the session.
- Collect a short post-session note and paper state.
- Generate a template SCORE from the session data.
- Store the generated SCORE in local storage.

Out of scope:
- Camera scan
- Sensor-based shake / rotate detection
- Real AI API integration
- Community sharing
- Payment / premium features
```

このチケットを実装した後、実際に紙を使って1分間の身体リサーチを行い、体験ログから次の仕様修正に入る。
